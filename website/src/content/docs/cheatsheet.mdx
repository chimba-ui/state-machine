Every term and concept in `@dunky-dev/state-machine`, with a one-line meaning and a link to its full doc page.

## Building blocks

| Term           | Meaning                                                                                                                                                                           |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **machine**    | The built service from `machine(config)`; exposes `start`/`stop`/`send`/`state`/`context`/`select`. [â†’](/api/setup)                                                              |
| **setup**      | The authoring entry point: `setup.infer().createMachine(literal)` (types inferred) or `setup.as<Ctx,Ev>().config(registries).createMachine(config)` (names compile-checked). [â†’](/api/setup) |
| **state**      | One of the flat, named situations the machine can be in (exactly one at a time). [â†’](/api/states)                                                                                 |
| **transition** | An `on` entry: where an event takes the machine (optional `target`, `guard`, `actions`). [â†’](/api/states)                                                                         |
| **event**      | The `{ type, â€¦ }` object you `send()` to drive a transition. [â†’](/api/states)                                                                                                     |
| **context**    | The machine's data: one plain object, read directly (`m.context.x`), written via `setContext`. [â†’](/api/context)                                                                  |
| **setContext** | The single, batched entry point for writing context (shallow-equal deduped). [â†’](/api/context)                                                                                    |
| **send**       | Dispatch an event to the machine; events run to completion. [â†’](/api/states)                                                                                                      |

## Transitions & actions

| Term                  | Meaning                                                                                                                               |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **guard**             | A predicate that gates a transition: return `false` and it doesn't fire. [â†’](/api/guards)                                            |
| **and / or / not**    | Guard combinators for composing named guards. [â†’](/api/guards)                                                                        |
| **fallthrough**       | An array of transitions for one event; the first whose guard passes wins. [â†’](/api/guards)                                            |
| **action**            | A fire-and-forget side-effect run on a transition, in order; gets `{context, setContext, event, send, computed}`. [â†’](/api/actions)  |
| **act**               | Write-sugar returning a context-writing action: `act({ field: value })` instead of the `setContext` wrapper. [â†’](/api/actions)       |
| **oneOf**             | Conditional action: variadic `{ guard?, actions }` branches, first passing wins (the action analog of fallthrough). [â†’](/api/actions) |
| **entry / exit**      | Actions run when a state is entered / left (any path in or out). [â†’](/api/actions)                                                    |
| **run-to-completion** | Events queue: a `send()` inside an action waits until the current transition finishes. No re-entrancy. [â†’](/api/states)              |

## Time, data & derivation

| Term         | Meaning                                                                                                                         |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| **after**    | A timed transition: fires after a delay while in a state; auto-cancelled on exit. [â†’](/api/timers)                             |
| **delay**    | An `after` key: a number of ms, or a named delay from `implementations.delays` (can read context). [â†’](/api/timers)             |
| **watch**    | Run actions whenever a context/computed field changes, in any state, for the machine's lifetime. [â†’](/api/watch)               |
| **computed** | A lazy, memoized value derived from context (or other computeds); recomputes only when a read input changes. [â†’](/api/computed) |

## Effects & the platform seam

| Term                | Meaning                                                                                                                                      |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **effect**          | A side-effect with cleanup, scoped to a state: runs on enter, its returned cleanup runs on exit. [â†’](/api/effects)                           |
| **ComponentEffect** | A platform-specific effect declared outside the machine; touches the DOM or reads props, calls `send()` into the machine. [â†’](/api/effects) |
| **implementations** | The named registry on a config: `guards` / `actions` / `effects` / `delays` referenced by string. [â†’](/api/guards)                          |

## The view boundary

| Term                             | Meaning                                                                                                                                    |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **connect**                      | A pure function mapping a machine snapshot â†’ the view-facing api (handlers + attributes). [â†’](/api/connector)                              |
| **connector**                    | Keeps `connect` live: memoizes the snapshot, makes props a reactive input, wires reactions. [â†’](/api/connector)                            |
| **snapshot**                     | The memoized view api the connector serves; stable identity until the machine or props change. [â†’](/api/connector)                        |
| **setProps**                     | Push new props into the connector (a reactive input; shallow-dedup'd). [â†’](/api/connector)                                                 |
| **reaction**                     | A `[selector, callback]` tuple that fires a prop-callback from _outside_ the machine on a value change. [â†’](/api/reactions)                |
| **makeReaction**                 | Inference helper for a reaction tuple; recovers the selectorâ†’callback `Value` type link. [â†’](/api/reactions)                              |
| **bindings**                     | The agnostic event/attr vocabulary `connect` speaks: `onPress`, `role`, `describedBy`. [â†’](/api/connector)                                |
| **normalize**                    | The per-target step translating bindings â†’ real props (`onPress` â†’ `onClick`; `aria-*` on web). [â†’](/api/connector)                        |
| **the machine never sees props** | The defining rule: a machine is pure behavior; props live only at the edge. [â†’](/api/connector)                                            |
| **the edge**                     | Where props/platform meet the machine: the connector (reactions) + the target's `ComponentEffect` (platform listeners). [â†’](/api/effects) |

## Observing changes

| Term          | Meaning                                                                                                                       |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| **subscribe** | Coarse observation: fires on _any_ state/context change (what a `useSyncExternalStore` bridge uses). [â†’](/api/subscriptions) |
| **select**    | Fine-grained observation: narrows to a slice, fires only when _that value_ changes. [â†’](/api/subscriptions)                  |
| **selection** | What `select(...)` returns: a value-deduped view with `.value` + `.subscribe`. [â†’](/api/subscriptions)                        |

## Composition & scale

| Term            | Meaning                                                                                                                         |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **compose**     | Run several peer machines as one unit (orthogonal regions): the answer to "nested/parallel" without nesting. [â†’](/api/compose) |
| **composition** | What `compose(...)` returns: bundled `start`/`stop` plus `sync` + `combine`. [â†’](/api/compose)                                  |
| **sync**        | A coarse cross-region rule on a composition; runs when any member changes. [â†’](/api/compose)                                   |
| **combine**     | A value-deduped selection derived across composition members. [â†’](/api/compose)                                                 |
| **tags**        | Labels on states so consumers query a _category_ (`hasTag('visible')`) instead of a name. [â†’](/api/states)                      |
| **hasTag**      | Check whether the current state carries a tag. [â†’](/api/states)                                                                 |
| **matches**     | Exact-state check: `m.matches('open')`. [â†’](/api/states)                                                                       |
| **createStore** | A tiny reactive cell (value + listeners) for singleton state _outside_ any one machine. [â†’](/api/create-store)                  |
| **store**       | What `createStore(...)` returns: `get` / `set` / `subscribe` (+ optional domain methods). [â†’](/api/create-store)                |
